### 알고리즘과 객체지향 설계
* 알고리즘의 각 단계를 책임 관점에서 여러 객체로 분배한다.
* 책임 할당
  * 알고리즘을 구성하는 로직을 적합한 객체에게 분배하는 일을 객체지향에서는 책임 할당이라고 부른다.

* 협력은 애플리케이션 기능이라는 문맥 안에서 얻고 책임은 협력이라는 문맥안에서 얻는다면 책임을 맡길 객체는 어디에서 얻을 수 있을까?

### 도메인 모델
* 도메인에 중요한 개념과 관계의 집합
* 객체지향 설계의 재료로 사용되는 도메인의 추상화
* 후보 객체를 찾을 때 가장 먼저 참고하는 개념과 관계의 집합
* 도메인 모델 안에 포함된 개념과 관계는 객체지향 설계에서 책임을 할당받을 객체의 이름과 관계에 대한 중요한 힌트를 제공한다.
* 추상화는 의도적으로 우리가 사용할 용도에 적합한 부부만 취하고 나머지는 버리는 단순화 과정을 의미한다.

### 표현적 차이
* 도메인에 대한 개념적 모델과 소프트웨어 구현 사이의 거리
* 도메인에 모습과 코드의 모습이 다르면 표현적 차이가 크다

### 표현적 차이를 줄이는 이유
* 변경하기 쉽고 유연한 설계를 얻을 수 있다.

### 안정적인 도메인
* 도메인의 개념과 기본 구조는 비즈니스가 바뀌지 않는 이상 그대로 유지

### 안정적인 도메인에 기반한 안정적인 코드
* 안정적인 도메인의 구조에 기반한 코드 구조 역시 변경에 안정적이다.
* 요구사항이 변경되더라도 코드의 전체적인 구조는 유지된다.

### 은유를 통한 개념의 연상
* 은유는 책임을 수행하는데 적합한 객체를 도메인 개념과 연결한다.
* 코드의 위치를 파악하기 용이하다.
* 도메인의 개념을 은유해서 객체와 클래스의 이름을 지으면 책임을 구현한 코드를 찾기가 쉬워진다.
* 수정할 코드의 위치를 파악하기 용이하다.

### 명사의 가치
* 객체지향은 명사를 이용해 동사를 묶을 수 있는 개념적 틀 제공한다.
  * 명사를 이용해서 동사 묶기
    * 요구사항이 변경될 때 어떤 클래스를 수정할지 쉽게 찾을 수 있다.
    * 동사(책임) -> 명사(도메인 개념)    명사+동사(클래스)
    * 할인 여부를 판단한다. -> 할인 조건

### GRASP
* 일반적인 책임 할당 소프트웨어 패턴 General Responsibility Assignment Software Pattern
* 9가지 패턴
  * 창조자
  * 정보 전문가
  * 낮은 결합도
  * 높은 응집도
  * 컨트롤러
  * 간접화
  * 다형성
  * 변경 보호
  * 순수한 가공물

### CRC 카드
* 책임과 협력을 표현하기 위핸 객체지향 설계 도구
* Candidate : 후보 역할 또는 객체
* Responsibility : 책임
* Collaborator : 협력자

### CRC는 객체 협력을 위한 설계 도구
* 정적인 클래스가 아닌 실제로 행동을 수행하는 동적인 객체를 의미한다. (런타임)
* 책임을 다루기 때문에 런타임 객체 협력을 설계하는 용도로 사용된다.
* 클래스와 유사해보이지만 아니다.
