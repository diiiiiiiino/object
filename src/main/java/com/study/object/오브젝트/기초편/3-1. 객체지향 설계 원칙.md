### 객체지향의 핵심은 변경관리

### 변경하기 쉬운 객체지향 설계 원칙/ 순서
1. 협력에 필요한 행동을 먼저 결정하고 행동에 적합한 객체를 나중에 선택하라 (객체 사이의 협력)
    * 고립된 상태에서 클래스의 행동을 결정하는 것은 불가능하다.
    * 협력에 필요한 요청에서 행동을 결정해야 한다.
    * 코드를 수정하지 않고도 다양한 객체와 협력 가능
    * 행동을 먼저 결정하기 때문에 행동을 수행할 수 있는 객체라면 협력에 참여 가능

2. 객체의 행동을 먼저 구현하고 행동에 필요한 데이터를 나중에 결정하라 (클래스 내부 구현)
  * 데이터를 고려하지 않고 외부에 어떤 행동을 제공할 지 먼저 결정해라
  * 메서드를 구현한다.
  * 메서드를 구현하는데 필요한 데이터 추가
  * 행동을 선택하는 기준은?
    * 외부 객체의 요청에 응답하기 위해 필요
  * 코드르르 수정하지 않고도 데이터 변경 가능
  

### 협력
* 진정한 객체지향 설계는 제한된 문맥 안에서 설계한다.
* 협력이 하나의 객체 설계를 위한 문맥 제공한다.

### 객체지향 설계 순서
* 객체 협력(런타임)에서 클래스 구조(컴파일타임)의 순서로 진행
