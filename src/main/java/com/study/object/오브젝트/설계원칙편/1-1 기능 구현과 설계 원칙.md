# 🧠 설계에 대한 핵심 정리 노트

## 1. 프로그래밍은 두 가지 관점으로 본다

프로그래밍은 하나의 관점으로 보면 항상 부족하다.
**반드시 두 관점을 동시에 고려해야 한다.**

### 1️⃣ 기능 관점 (Function View)

* 시스템이 **외부에 제공하는 행동**
* 사용자가 실제로 사용하는 기능
* 개발자가 코드를 작성하는 가장 큰 이유

**목표**

* 유용한 기능을
* 버그 없이
* 정확하게 구현하는 것

> 기능 관점의 클라이언트는 **사용자**다.

---

### 2️⃣ 구조 관점 (Structure View)

* 기능을 구현하기 위해 **코드를 어떤 구조로 배치할 것인가**
* 시간이 지나도 코드를 읽고 수정할 수 있어야 한다

**목표**

* 다른 개발자가
* 쉽게 이해하고
* 안전하게 수정할 수 있는 코드

> 구조 관점의 클라이언트는 **미래의 개발자**다.

---

## 2. 개발자는 두 명의 클라이언트를 동시에 상대한다

| 관점 | 클라이언트 | 중요 가치       |
| -- | ----- | ----------- |
| 기능 | 사용자   | 지금 잘 동작하는가  |
| 구조 | 개발자   | 변경에 잘 대응하는가 |

👉 **설계의 가치는 구조 관점에서 드러난다.**

---

## 3. 설계란 무엇인가

설계는 기능을 구현하는 일이 아니다.

> **설계란
> 어떤 기능을 구현하기에
> 적절한 코드 구조를 결정하는 작업이다.**

다시 말하면:

* 유지보수하기 쉽게
* 변경 가능하도록
* 코드를 배치하는 방법을 정하는 일

---

## 4. 객체지향에서 기능은 어떻게 구현되는가

객체지향 패러다임에서는 기능을 **객체 하나가 아니라 협력으로 구현**한다.

### 객체지향의 기본 관점

* 기능 → **객체 사이의 메시지**
* 구현 → **객체의 책임**
* 구조 → **객체들의 협력 관계**

---

## 5. 객체지향 설계의 흐름

1. 사용자에게 제공할 기능을 정의한다
2. 기능을 구성하는 **행동을 책임으로 분해한다**
3. 책임을 **객체에게 할당한다**
4. 객체들이 메시지를 주고받으며 협력하도록 만든다
5. 이 협력을 코드 구조로 고정한다

> 클래스는 출발점이 아니라 **결과물**이다.

---

## 6. 책임 주도 설계 (Responsibility-Driven Design)

책임 주도 설계는 객체지향 설계의 대표적인 방법이다.

### 책임 주도 설계의 핵심

* 시스템의 기능을 **책임의 관점**으로 해석한다
* 책임을 협력 단위로 나눈다
* 책임을 객체에 할당해 구조를 만든다

👉 가장 어렵고 중요한 부분은
**책임을 어떻게 나누고 누구에게 줄 것인가**다.

---

## 7. 설계가 어려운 이유

### 이유 1️⃣ 정답이 없다

* 동일한 기능도 설계는 무수히 많다
* 개발자 10명 → 설계 10개

### 이유 2️⃣ 변경은 예측할 수 없다

* 각 설계는 **설계자가 가정한 변경**에 최적화되어 있다
* 다른 변경이 오면 언제든 부적합해진다

> 설계는 기술 문제가 아니라 **판단의 문제**다.

---

## 8. 책임 주도 설계에 대한 오해

❌ 책임 주도 설계를 하면
→ 처음부터 모든 변경을 수용할 수 있는 안정적인 구조를 만들 수 있다

⭕ 실제로는
→ **변경을 조금 더 안정적으로 받아들일 가능성이 높아질 뿐**이다

* 미래를 예측하는 설계는 불가능하다
* 요구사항 변화는 필연적이다

---

## 9. 우리가 선택할 수 있는 유일한 전략

```
1. 현재 시점에서 가장 단순한 설계를 선택한다
2. 실제 변경이 발생하면
3. 변경에 맞게 구조를 개선한다
```

이 전략의 이름이 **리팩터링**이다.

---

## 10. 리팩터링의 정의

> **겉으로 보이는 동작은 유지한 채
> 내부 구조를 개선하는 작업**

* 기능 변경 ❌
* 구조 개선 ⭕

---

## 11. 설계 원칙의 역할

설계 원칙은 **절대적인 법칙이 아니다.**

설계 원칙은:

* 코드 품질을 판단하는 기준
* 어디가 문제인지 알려주는 힌트
* 리팩터링 방향을 제시하는 가이드

> “이 구조, 지금도 괜찮은가?”를 묻기 위한 도구

---

## 12. 책임 주도 설계와 설계 원칙의 관계

* 책임 주도 설계 → **구조를 만드는 방법**
* 설계 원칙 → **구조를 평가하는 기준**

설계 원칙을 고려하면:

* 책임의 크기를 조절하기 쉽고
* 객체 간 협력 구조를 더 명확히 할 수 있다

---

## 13. 이 내용을 기억하기 위한 핵심 문장

* 프로그래밍에는 **두 관점**이 있다
* 개발자는 **두 명의 클라이언트**를 상대한다
* 설계는 **구조를 다루는 일**이다
* 객체지향은 **협력**이다
* 책임 주도 설계는 **변경 완화 전략**이다
* **단순하게 설계하고, 변경되면 리팩터링한다**

---

## 14. 나만의 설계 체크 질문

* 이 책임은 너무 크지 않은가?
* 이 객체는 혼자 너무 많은 걸 알고 있지 않은가?
* 변경이 오면 어디가 가장 먼저 흔들릴까?
* 지금 구조가 가장 단순한가?
* 리팩터링 방향이 보이는가?
